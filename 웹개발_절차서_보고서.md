## 1. 개요

### 1.1 프로젝트 소개

| 항목 | 내용 |
|------|------|
| 앱 이름 | 월 지출관리 |
| 현재 버전 | V2.3_Web |
| 웹 주소 | https://monthly-expenses-navy.vercel.app/ |
| GitHub 저장소 | jengking99-collab/monthly-expenses |
| 주요 기술 스택 | React 19, Vite 7, Firebase Firestore, Vercel |

이 문서는 월 지출관리 웹 버전의 개발·배포·유지보수를 위한 전체 절차를 기술합니다.  
GitHub(코드 저장) → Vercel(자동 빌드/배포) → Firebase(데이터 동기화) 3개 서비스를 연동하여  
코드 한 번 push로 전 세계 어디서나 접속 가능한 웹 앱을 운영합니다.

### 1.2 사용 서비스 요약

| 서비스 | 역할 | 플랜 | 월 비용 |
|--------|------|------|---------|

| **Vercel** | 웹 앱 자동 빌드·배포·CDN 호스팅 | Hobby | 무료 |
| **Firebase** | 기기 간 지출 데이터 실시간 동기화 | Spark | 무료 |

## 2. 서비스 소개 및 역할

### 2.1 GitHub — 코드 버전 관리

**정의**: 소스 코드의 유일한 원본(Single Source of Truth)을 저장하고 변경 이력을 추적하는 플랫폼입니다.

**핵심 역할**:

| 기능 | 설명 |
|------|------|
| 코드 저장 | 모든 소스 코드를 원격 서버에 안전하게 보관 |
| 버전 이력 | `git commit`마다 스냅샷 기록 → 언제든 과거로 되돌림 가능 |
| 브랜치 관리 | `master`(운영), `feature/*`(개발) 브랜치 분리 |
| 배포 트리거 | `master` push 시 Vercel Webhook 자동 호출 |
| 협업 | 코드 리뷰, 이슈 트래킹, PR 관리 |

**브랜치 전략**:
```
master  ──────●──────●──────●──────   (운영, 자동 배포됨)
               \            /
feature/xxx     ●──●──●────            (기능 개발, 미리보기 URL)
```

**이 프로젝트에서의 적용**:
- 단일 파일 앱(`src/App.jsx`) 변경 이력 전체 보관
- Vercel이 GitHub에서 코드를 pull하여 빌드 실행
- 긴급 롤백 시 이전 커밋으로 되돌린 후 push

---

### 2.2 Vercel — 웹 배포 및 호스팅

**정의**: GitHub의 코드를 자동으로 빌드하여 전 세계 CDN에 배포하는 클라우드 플랫폼입니다.

**핵심 역할**:

| 기능 | 설명 |
|------|------|
| 자동 CI/CD | GitHub push → 자동 빌드 → 자동 배포 |
| 글로벌 CDN | 전 세계 엣지 서버 캐싱 → 어디서나 빠른 로드 |
| 환경변수 관리 | Firebase 설정값 등 민감 정보를 안전하게 주입 |
| 미리보기 URL | `feature/*` 브랜치 → 별도 preview URL 자동 생성 |
| 배포 롤백 | 대시보드에서 이전 배포로 즉시 전환 |
| HTTPS 자동 | SSL 인증서 자동 발급·갱신 |

**빌드 설정**:

| 항목 | 설정값 |
|------|--------|
| Framework Preset | Vite |
| Build Command | `npm run build` |
| Output Directory | `dist` |
| Install Command | `npm install` |
| Node.js 버전 | 20.x |

**배포 흐름 (소요 시간)**:
```
git push → GitHub Webhook → Vercel 빌드 시작 (~10초)
         → npm install (~20초) → npm run build (~20초)
         → CDN 엣지 전파 (~30초) → 배포 완료
총 소요: 약 60~90초
```

---

### 2.3 Firebase — 데이터베이스 및 기기 간 동기화

**정의**: Google이 제공하는 BaaS(Backend as a Service)로, 서버 없이 클라우드 데이터베이스를 사용합니다.

**핵심 역할**:

| 기능 | 설명 |
|------|------|
| Firestore DB | NoSQL 문서 데이터베이스 (JSON 형태 저장) |
| 기기 간 동기화 | 모바일에서 입력 → PC에서 자동 반영 |
| syncKey 인증 | 사용자 정의 비밀 키로 본인 데이터만 접근 |
| 오프라인 내성 | Firebase 장애 시 로컬 localStorage로 계속 동작 |

**데이터 구조**:
```
Firestore
└── sync/
    └── {syncKey}/              ← 사용자가 설정한 비밀 키 (예: my_secret_key)
        ├── data: string        ← 월별 지출 전체 데이터 (JSON 문자열)
        ├── fixed_base: string  ← 고정항목 버전 이력 (JSON 문자열)
        └── updatedAt: timestamp
```

**동기화 방향 및 트리거**:

| 방향 | 트리거 | 주기 |
|------|--------|------|
| 앱 → Firestore (push) | 데이터 변경 감지 | 변경 후 5초 debounce 자동 전송 |
| Firestore → 앱 (pull) | 앱 로드 시 자동 | 매 시작 시 silent pull |
| Firestore → 앱 (수동) | 사이드바 동기화 버튼 클릭 | 사용자 요청 시 |

**무료 Spark 플랜 한도** (개인 사용 시 충분):

| 항목 | 무료 한도 | 실제 사용량 |
|------|----------|-----------|
| 읽기 | 50,000건/일 | ~100건/일 |
| 쓰기 | 20,000건/일 | ~50건/일 |
| 저장 용량 | 1 GiB | ~1 MB |

## 3. 서비스 간 상호 동작 원리

### 3.1 배포 파이프라인 상세 흐름

```
개발자 PC       GitHub          Vercel           사용자 브라우저
    │              │               │                    │

    ├─────────────►│               │                    │
    │              │  Webhook POST │                    │
    │              ├──────────────►│                    │
    │              │               │ git clone/pull     │
    │              │◄──────────────┤ 코드 다운로드       │
    │              │               │                    │
    │              │               │ npm install        │
    │              │               │ npm run build      │
    │              │               │ → dist/ 생성       │
    │              │               │                    │
    │              │               │ CDN 엣지 전파      │
    │              │               ├───────────────────►│ HTML/JS/CSS
    │              │               │                    │ 다운로드
    │              │               │                    │
```

### 3.2 데이터 동기화 흐름 (기기 A → 기기 B)

```
[기기 A: 모바일]        [Firebase Firestore]         [기기 B: PC]
        │                        │                         │
        │ 지출 항목 입력          │                         │
        │ (5초 후 자동 push)      │                         │
        ├───────────────────────►│                         │
        │                        │ sync/{key} 문서 업데이트 │
        │                        │                         │
        │                        │      기기 B 앱 시작      │
        │                        │◄────────────────────────┤
        │                        │ auto pull               │
        │                        ├────────────────────────►│
        │                        │ 최신 데이터 전달          │
```

### 3.3 환경변수 주입 경로

Firebase 설정값(API Key 등)은 두 경로로 주입됩니다:

```
Firebase Console (SDK 설정값)
        │
        ├──► 개발 환경: .env.local  ← gitignore, 로컬 전용
        │         └── npm run dev 시 VITE_* 변수로 번들에 주입
        │
        └──► 운영 환경: Vercel 환경변수 설정
                  └── npm run build 시 VITE_* 변수로 dist/ 번들에 주입
                       └── monthly-expenses-navy.vercel.app 에서 동작
```

> ⚠️ **보안 원칙**: `.env.local`은 절대 git에 커밋하지 않습니다.  
> Vercel 대시보드에 별도 등록해야 운영 환경에서 Firebase가 동작합니다.

### 4.3 환경변수 설정 (.env.local)

프로젝트 루트에 `.env.local` 파일을 생성하고 Firebase 설정값을 입력합니다.  
설정값은 **Firebase Console → 프로젝트 설정 → 내 앱 → SDK 구성**에서 복사합니다.

```env
# Firebase 설정 — Firebase Console에서 복사
VITE_FIREBASE_API_KEY=여기에_API_KEY_입력
VITE_FIREBASE_AUTH_DOMAIN=프로젝트ID.firebaseapp.com
VITE_FIREBASE_PROJECT_ID=프로젝트ID
VITE_FIREBASE_STORAGE_BUCKET=프로젝트ID.firebasestorage.app
VITE_FIREBASE_MESSAGING_SENDER_ID=숫자ID
VITE_FIREBASE_APP_ID=앱ID
```

> `VITE_` 접두사는 필수입니다. Vite는 이 접두사가 있는 변수만 클라이언트 번들에 포함합니다.

## 5. 서비스 초기 설정 절차 (신규 프로젝트 구성 시)

> 이미 구성된 프로젝트를 클론하는 경우 4장만 참고하면 됩니다.  
> 이 장은 처음부터 동일한 구성을 만들 때 사용합니다.

### 5.1 GitHub 초기 설정

**Step 1**: github.com 접속 → 계정 생성 또는 로그인

**Step 2**: 새 저장소 생성
- `+ New repository` 클릭
- Repository name 입력 (예: `monthly-expenses`)
- `Private` 또는 `Public` 선택
- `Create repository`

**Step 3**: 로컬 저장소 연결
```bash
git init
git add -A
git commit -m "feat: 초기 프로젝트 구성"
git branch -M master
git remote add origin https://github.com/{계정}/{저장소명}.git
git push -u origin master
```

**Step 4**: `.gitignore` 확인
```
node_modules/
dist/
.env.local      ← Firebase 설정값 보호를 위해 필수
dev.log
releases/
```

---

### 5.2 Vercel 초기 설정

**Step 1**: vercel.com 접속 → GitHub 계정으로 로그인

**Step 2**: 프로젝트 가져오기
- `Add New... → Project` 클릭
- GitHub 저장소 목록에서 `monthly-expenses` 선택
- `Import`

**Step 3**: 빌드 설정 확인 (Vite 자동 감지됨)

| 항목 | 값 |
|------|-----|
| Framework Preset | Vite |
| Root Directory | `./` |
| Build Command | `npm run build` |
| Output Directory | `dist` |

**Step 4**: 환경변수 등록 (중요!)
- `Environment Variables` 섹션에서 `.env.local`의 모든 변수 등록
- Name: `VITE_FIREBASE_API_KEY` / Value: 실제 값 입력
- 동일한 방법으로 6개 변수 모두 등록

**Step 5**: `Deploy` 클릭 → 첫 배포 완료 확인  
- 배포 URL 자동 생성 (예: `monthly-expenses-navy.vercel.app`)

---

### 5.3 Firebase 초기 설정

**Step 1**: console.firebase.google.com → Google 계정 로그인

**Step 2**: 새 프로젝트 생성
- `프로젝트 추가` 클릭
- 프로젝트 이름 입력 (예: `monthly-expenses`)
- Google 애널리틱스: 선택 해제 후 `프로젝트 만들기`

**Step 3**: Firestore 데이터베이스 생성
- 왼쪽 메뉴 `빌드 → Firestore Database`
- `데이터베이스 만들기`
- **프로덕션 모드** 선택
- 리전: `asia-northeast3 (Seoul)` 권장

**Step 4**: 보안 규칙 설정
- Firestore → `규칙` 탭 → 아래 내용으로 수정 후 `게시`

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /sync/{syncKey} {
      allow read, write: if true;
    }
  }
}
```

> 보안은 syncKey의 비공개에 의존합니다. 타인에게 syncKey를 공유하지 마세요.

**Step 5**: 앱 등록 및 SDK 설정값 획득
- 프로젝트 설정(⚙️) → `내 앱` 탭 → `앱 추가` → 웹(`</>`) 선택
- 앱 닉네임 입력 → `앱 등록`
- `firebaseConfig` 객체의 각 값을 `.env.local` 형식으로 변환하여 저장

## 6. 개발 및 배포 절차

### 6.1 일반 코드 수정 → 배포 (일상 워크플로우)

```
① 코드 수정
   └── 주로 src/App.jsx 편집 (단일 파일 구조)

② 로컬 테스트
   └── npm run dev → http://localhost:5173 에서 기능 확인

③ 버전 업데이트 (배포 시마다)
   ├── package.json → "version": "2.4.0" (예: 다음 버전)
   └── App.jsx 사이드바 → V2.4_Web 문자열 수정

④ 커밋
   └── git add src/App.jsx package.json
       git commit -m "feat(v2.4): 기능 설명"

⑤ 배포
   └── git push origin master

⑥ 배포 확인 (약 60~90초 후)
   └── vercel.com 대시보드 → Deployments → 최신 배포 상태 확인
       https://monthly-expenses-navy.vercel.app/ → 버전 문자열 확인
```

### 6.2 브랜치를 활용한 개발 절차

```bash
# 기능 개발 시작
git checkout -b feature/새기능이름
# ... 코드 수정 및 로컬 테스트 ...
git add -A
git commit -m "feat: 새 기능 설명"
git push origin feature/새기능이름
# → Vercel에서 미리보기 URL 자동 생성됨 (테스트 가능)

# 검토 완료 후 master 배포
git checkout master
git merge feature/새기능이름
git push origin master        # 운영 배포 트리거
```

### 6.3 배포 확인 방법

| 확인 항목 | 확인 방법 |
|----------|---------|
| 빌드 성공 여부 | Vercel 대시보드 → Deployments → 상태 아이콘 |

| Firebase 연결 | 사이드바 동기화 상태 뱃지 확인 |

## 7. 유지보수 절차

## 8. 백업 절차

### 8.1 코드 백업 (GitHub 활용)

코드는 GitHub에 항상 최신 원본이 보관됩니다.

```bash
# 최신 코드 로컬에 동기화
git pull origin master

# 버전 이력 확인
git log --oneline -10

# 특정 버전의 코드 확인
git show {커밋해시}:src/App.jsx

# 긴급 롤백 (운영 배포 되돌리기)
git revert {되돌릴커밋해시}   # 안전 방법 — 새 커밋으로 이전 상태 복원
git push origin master        # → Vercel 재배포 트리거
```

**Vercel 대시보드 롤백 (가장 빠른 방법)**:
1. vercel.com → 해당 프로젝트 → `Deployments` 탭
2. 되돌릴 이전 배포 선택 → `···` 메뉴 → `Promote to Production`
3. 약 10초 내 즉시 반영

### 8.2 Firebase 데이터 백업

**방법 1: 앱 내 JSON 내보내기 (권장, 가장 간편)**
1. 웹 앱 접속 → 사이드바 하단
2. `📤 전체 내보내기(JSON)` 클릭
3. 다운로드된 `.json` 파일 안전한 위치에 보관 (PC, 클라우드 드라이브 등)

**방법 2: Firebase Console에서 직접 확인**
1. Firebase Console → Firestore → `데이터` 탭
2. `sync/{syncKey}` 문서 내용 수동 복사

**복원 방법** (앱 내):
1. 웹 앱 접속 → 사이드바 → `📥 전체 가져오기(JSON)` 클릭
2. 백업한 `.json` 파일 선택 → 확인 대화상자에서 `확인`
3. Firebase 동기화 버튼 눌러 클라우드에 반영

### 8.3 신규 PC로 환경 이전 절차

```bash
# 1. 신규 PC에 Node.js, Git 설치

# 2. 저장소 클론
git clone https://github.com/jengking99-collab/monthly-expenses.git
cd monthly-expenses

# 3. 의존성 설치
npm install

# 4. .env.local 생성
# 기존 PC의 .env.local 내용 복사 또는 Firebase Console에서 새로 복사

# 5. 개발 서버 확인
npm run dev   # http://localhost:5173

# 6. Firebase 동기화 연결
# 웹 앱 → 사이드바 → 동기화 → 기존과 동일한 syncKey 입력 → pull
```

### 8.4 서비스 장애 시 대응

| 장애 서비스 | 앱에 미치는 영향 | 대응 방법 |
|------------|---------------|---------|
| **GitHub** | 코드 push 불가 (앱 동작에 영향 없음) | 로컬 커밋 유지, 복구 후 push |
| **Vercel** | 웹 앱 접속 불가 | vercel.com/status 확인, 이전 배포로 롤백 |
| **Firebase** | 기기 간 동기화 불가 (앱 자체는 동작) | 로컬 localStorage 데이터 유지, 복구 후 재동기화 |

> **Firebase 장애 시 데이터 보호**: 앱은 localStorage에 데이터를 실시간 저장하므로  
> Firebase 연결이 끊겨도 **데이터 손실 없이** 로컬 모드로 계속 사용 가능합니다.

## 9. 문제 해결 (트러블슈팅)

| 증상 | 원인 | 해결 방법 |
|------|------|---------|
| Vercel 빌드 실패 "VITE_FIREBASE_API_KEY is not defined" | Vercel 환경변수 미등록 | Vercel → Settings → Environment Variables → Firebase 변수 모두 등록 |
| 로컬에서 Firebase 오류 | `.env.local` 없거나 변수명 오류 | `.env.local` 파일 생성, `VITE_` 접두사 확인 |
| 동기화가 안 됨 | 기기 간 syncKey 불일치 | 모든 기기에서 동일한 syncKey 사용 확인 |
| Vite 8 업그레이드 후 빌드 실패 | Rolldown이 `firebase/firestore` subpath 처리 불가 | `package.json`에서 `"vite": "^7.0.0"`으로 다운그레이드 |
| 배포 후 구버전이 보임 | 브라우저 캐시 | 강력 새로고침: Ctrl+Shift+R (Windows) / Cmd+Shift+R (Mac) |
| 앱 로드 시 동기화 충돌 | 로컬과 클라우드 동시 변경 | 사이드바 동기화 버튼으로 수동 pull → 최신 데이터 확인 후 재입력 |
